# 归档目录标准

## 目录结构

```
archive/
├── backend/          # 后端归档文件
│   ├── deprecated/   # 已弃用但保留参考的代码
│   ├── migrated/     # 已迁移到新架构的旧代码
│   └── obsolete/     # 完全过时可删除的代码
├── frontend/         # 前端归档文件
│   ├── deprecated/   # 已弃用但保留参考的代码
│   ├── migrated/     # 已迁移到新架构的旧代码
│   └── obsolete/     # 完全过时可删除的代码
├── docs/             # 文档归档
│   ├── deprecated/   # 过时文档
│   ├── migrated/     # 已合并的文档
│   └── obsolete/     # 可删除的文档
└── scripts/          # 脚本归档
    ├── deprecated/   # 旧脚本
    ├── migrated/     # 已重构的脚本
    └── obsolete/     # 废弃脚本
```

## 归档规则

### 1. 归档时机
- 代码完成迁移后立即归档
- 功能被新方案替代时归档旧实现
- 文档内容过时或已合并时归档

### 2. 归档命名规范
- 文件名格式: `{原文件名}_archived_{YYYY-MM-DD}.{扩展名}`
- 示例: `old_service.py_archived_2026-02-23.py`

### 3. 归档说明文件
每个归档文件应附带说明文件，记录：
- 归档原因
- 替代方案
- 预计删除时间
- 负责人

### 4. 保留期限
- `deprecated/`: 保留6个月
- `migrated/`: 保留3个月
- `obsolete/`: 保留1个月后可删除

### 5. 定期清理
- 每月第一周检查归档目录
- 清理超过保留期限的文件
- 更新归档清单
